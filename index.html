Pregunta 1
¿Qué datos necesita el sistema para funcionar?
El sistema requiere tres capas de datos:

  • Datos de lugares: nombre, tipo/categoría, latitud y longitud reales, descripción multiidioma, costo estimado, horarios de atención, calificación promedio, dirección y reseñas representativas.
  • Datos del usuario: tipo de perfil (turista, estudiante, etc.), idioma preferido, rango de presupuesto, historial de lugares visitados y preferencias de interés.
  • Datos geoespaciales: tiles de mapa base (OpenStreetMap), coordenadas GPS verificadas, y contexto territorial para proyectar los marcadores con precisión.

En este prototipo los datos de lugares se almacenan como JSON local dentro del HTML, eliminando la dependencia de servidores externos y permitiendo uso offline.
Pregunta 2
¿Dónde se podrían almacenar esos datos?
  • Prototipo local (actual): JSON embebido en HTML — sin backend, funciona completamente offline.
  • Base de datos relacional: PostgreSQL + extensión PostGIS para consultas espaciales eficientes (radio, polígonos, rutas más cercanas).
  • Base de datos NoSQL: Firebase Firestore o MongoDB Atlas para esquemas flexibles, sincronización en tiempo real y escalabilidad.
  • APIs externas: Google Places API u OpenStreetMap/Overpass API para datos enriquecidos y actualizados automáticamente en producción.
  • Perfil de usuario: LocalStorage (sesión local) para prototipos, o JWT + backend autenticado para persistencia segura entre dispositivos.
Pregunta 3
¿Cómo ayudaría la IA o el análisis de datos a mejorar esta aplicación?
  • Recomendación personalizada: Modelos de filtrado colaborativo que aprenden del historial de visitas y preferencias de perfiles similares para sugerir lugares relevantes de forma automática.
  • Itinerarios inteligentes: Algoritmos de optimización de ruta (TSP/greedy) que generen recorridos según tiempo disponible, presupuesto y tipo de experiencia deseada.
  • Análisis de sentimiento: NLP sobre reseñas de Google Maps para generar calificaciones dinámicas y alertas tempranas de deterioro en la calidad del servicio.
  • Predicción de afluencia: Modelos de series de tiempo que indiquen qué tan concurrido estará un lugar según hora, día y temporada, ayudando al usuario a planificar mejor.
  • Búsqueda semántica: LLMs integrados que interpreten consultas en lenguaje natural: "¿dónde puedo almorzar económico con niños hoy cerca del malecón?"
Pregunta 4
¿Qué riesgo de privacidad podría existir?
  • Geolocalización sin consentimiento: Acceder a la posición GPS del usuario sin opt-in explícito viola la Ley 1581/2012 de Colombia (habeas data) y normativas internacionales como el GDPR.
  • Construcción de perfil sensible: Combinar edad, rol, historial de rutas y lugares visitados permite inferir condición médica, creencias religiosas, hábitos o situación económica del usuario.
  • Transmisión a terceros: El uso de tiles de OpenStreetMap o APIs de Google implica enviar metadatos de navegación a esos proveedores, quienes tienen sus propias políticas de uso de datos.
  • Almacenamiento inseguro: Guardar preferencias y ubicaciones en LocalStorage sin cifrado las expone a scripts maliciosos mediante ataques XSS.
  • Mitigación recomendada: Anonimización de datos, consentimiento informado granular, cifrado en tránsito (HTTPS/TLS), política de retención mínima y opción de borrado completo del perfil.
